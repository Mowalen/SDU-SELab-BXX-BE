# 实验三  软件过程模型、Scrum开发方法、可行性分析报告
## 实验目的：
1. 研讨传统软件开发过程模型与敏捷开发
2. 了解Scrum开发方法
3. 完成可行性分析报告
## 实验内容
### 小组分工讨论传统软件开发过程模型与敏捷开发（中几种主要方法）的比较，分析各自的优缺点，以及如何应用于自己的项目中？并且分析自己项目中可能存在的风险，细化风险管理（做出风险分级及应对预案）
#### 讨论传统软件开发过程模型与敏捷开发（中几种主要方法）的比较及优缺点
* 传统软件开发过程模型
    * 优点：
        1. 结构化流程：瀑布模型提供了一个清晰的阶段性工作流程，从需求分析、设计、编码、测试到维护，每个阶段都有严格的完成标准和验收条件。
        2. 文档详尽：在项目早期就制定了详细的规格说明书和设计文档，有助于确保所有参与者对项目目标和预期结果有统一理解。
        3. 易于管理和控制：在项目初期明确了需求和计划，便于监控进度和控制预算。
    * 缺点：
        1. 需求变更困难：一旦进入下一阶段，需求更改的成本非常高，因为各阶段间的依赖关系很强，不易调整方向。
        2. 缺乏灵活性：无法快速响应市场变化和技术更新，尤其是在电商环境中，用户需求可能会频繁变化，瀑布模型难以适应。
        3. 延迟反馈：测试通常在开发后期进行，如果前期设计阶段存在错误或遗漏，可能导致大量返工。
* 敏捷开发
    * 优点：
        1. 高度灵活性：敏捷开发强调快速响应变化，能够适应项目执行过程中不断变化的需求。
        2. 迭代开发：通过短周期的Sprint（冲刺）来实现产品的迭代交付，每个迭代都能产出可运行的产品版本，客户可以在早期就看到并验证成果。
        3. 持续集成与测试：在整个开发过程中都融入测试环节，降低缺陷积累的风险，保证产品质量。
        4. 客户参与：通过定期评审会议，客户或最终用户可以参与到开发过程中，提高产品满足实际需求的可能性。
    * 缺点：
        * 不确定性管理：虽然敏捷适应变化，但过度的灵活性可能导致项目总体架构或长期规划不够稳定。
        * 依赖团队成熟度：敏捷方法的成功实施需要高度自我组织、高效沟通和协同工作的团队。
        * 资源管理：如果没有合适的敏捷实践和工具支持，可能会面临资源分配、优先级排序等方面的挑战。
#### 如何应用到自己的项目于中
对于商品网上交易系统这类涉及众多业务逻辑和用户体验的项目，敏捷开发或许更加适合，因为它允许快速迭代和用户反馈，以优化功能和界面设计。同时，敏捷框架如Scrum可以帮助团队划分任务为短期可完成的用户故事，并通过持续集成和交付确保各个功能模块的稳定性。
#### 做出风险分级及应对预案
* 风险分级：
    1. 高风险：需求频繁变更导致开发混乱；关键功能模块的技术难题；安全性漏洞。
    2. 中风险：用户界面设计不符合用户期望；系统性能瓶颈；第三方服务集成失败。
    3. 低风险：文档更新滞后；团队成员流动性；小范围的功能调整
* 应对预案：
    1. 高风险：采用敏捷开发方法，建立有效的需求变更管理机制；对关键技术点提前研究，必要时引入外部专家；严格遵循安全开发规范，定期进行安全审计和渗透测试。
    2. 中风险：通过原型设计和用户测试验证UI/UX；制定性能基准并进行负载测试；选择成熟稳定的第三方服务，并设置备用方案。
    3. 低风险：提倡轻量级文档，实时同步更新；强化团队内部培训和知识共享，减少人员流动影响；合理安排迭代计划，预留缓冲时间处理较小的变更请求。
### 阅读Scrum开发方法文档，理解Scrum过程工作模型
* Scrum是一种用于产品或项目开发的敏捷框架，它通过迭代和增量的方式促进了跨职能团队的高效协作。相比于传统的单职能团队和瀑布式开发模式，Scrum通过缩短反馈循环、增强透明度和频繁调整，提高了适应变化的能力，并减少了风险，进而提升了项目的灵活性和投资回报。
* Scrum的核心组成部分包括：
    1. Sprint：Sprint是一个严格限制在1至4周（一般建议2周）的时间盒，不论工作是否完成都在预定日期结束。每个Sprint都致力于从产品待办事项列表（Product Backlog）中挑选优先级最高的需求进行开发，团队在Sprint初期通过Sprint计划会议确定要完成的目标，并在整个Sprint期间专注于实现这些目标。Sprint结束后，团队进行评审会议和回顾会议，展示已构建的功能，并收集反馈以应用于下一轮迭代。
    2. 角色：Scrum团队由三个核心角色构成——产品负责人（Product Owner）、开发团队和ScrumMaster。产品负责人负责产品价值最大化，维护产品待办事项列表，并持续对其优先级进行排序。开发团队是跨职能的，包括了实现产品增量所需的所有必要技能，自行决定Sprint中的工作量和完成方式，且每个成员都能根据需求承担多种职责。ScrumMaster则是团队的支持者和敏捷实践的推动者，帮助团队克服障碍、保持聚焦，并促进团队成长。
    3. 产品待办事项列表（Product Backlog）：这是一个动态的、按优先级排列的需求列表，产品负责人负责细化并持续更新它，确保每个事项都被适当地估算了工作量、价值和技术风险，并随着学习和变化而演变。
    4. 每日Scrum：团队每天进行短暂的站会检查进展、讨论计划，并解决潜在问题。
    5. 完成的定义：每个Sprint产生的输出是一个“潜在可交付产品增量”，这意味着团队需在Sprint内完成所有必要的活动以交付可部署的产品部分。团队和产品负责人需在Sprint开始前共同确定“完成”的标准，确保增量产品不仅满足功能需求，还经过了集成、测试和文档编写等环节，具备交付条件。
* Scrum的实施过程中强调“审视并调整”，鼓励团队以小步骤快速迭代，通过不断的反馈和反思调整产品方向和开发过程，以应对未知和变化。此外，Scrum强调团队自组织和自管理，反对过度的层级管理和微观管理，提倡团队成员之间的互助合作和多技能学习，以形成高效且稳定的“特性团队”。
* Scrum的成功实施依赖于对框架本身原则的深刻理解和严格执行，而不应在遇到挑战时随意改变Scrum的基本实践。团队的成长和发展也需要专业的培训和ScrumMaster的指导，而非简单地由上级管理者强加Scrum实践。应该在初次尝试Scrum后，根据实际体验和效果，自主决定是否继续采用Scrum作为开发方式。
